01

Branching fiction fails like software, not like prose

A novel can only be bad. A branching novel can be broken — and broken in ways no amount of rereading reliably catches. A scene nobody can reach. A room with a locked door and no key, because the key is granted two chapters later. An ending that exists in the file and in no possible playthrough. A choice gated on an item that a careful reader could never have picked up.

These are not literary failures. They are graph failures — dangling references, unreachable nodes, unsatisfiable preconditions. Software has known how to catch those for fifty years. The only thing standing in the way is that stories are usually written as prose, and prose is opaque to a checker.

So the whole architecture follows from one decision: keep the prose, but make the structure a first-class datum. A scene is a JSON node. A choice is an edge with a precondition. Once the story is a graph, the graph can be walked — every path, every time, on every save.

The load-bearing idea

The engine and the checkers read the same files. There is no separate "design database" that drifts out of sync with what ships — ch3.json is simultaneously the thing the player plays and the thing the verifier proves. Nothing can be true of one and false of the other.

02

Four layers, and a hard rule about which way they point

Each layer knows only about the one beneath it. Canon constrains content; content feeds the runtime; verification reads content and canon and answers to both. The runtime knows nothing about any particular story — which is precisely what lets four very different stories share it.

Canon
The story bible — world, characters, the flag registry, the ending map, the waypoint budget. Prose for humans, but IDs are law: if a chapter contradicts the bible, one of them is wrong and the conflict gets resolved deliberately. Deliberately gitignored — the design docs stay off the public repo.
story-*/BIBLE.md  ·  local only
Content
The story itself, as pure data. A manifest.json naming the chapters, the skills and the recap summaries; one chN.json per chapter holding a scene graph. No logic, no code — every branch, gate and item is a field.
manifest.json  ·  ch1…ch19.json  ·  stories.json
Runtime
One generic interpreter. It fetches a manifest, walks a scene graph, types the prose out character by character, resolves gates, and writes progress to localStorage. 1,103 lines of vanilla JS. No framework, no build step, no dependencies — it opens from a file path and will still run in a decade.
js/engine.js  ·  css/style.css  ·  index.html
Verification
The compiler for the story. validate.py statically analyses the graph — references, reachability, gate feasibility, canon conformance. simulate.py exhaustively plays it — every reachable combination of scene and inventory — and proves the endings can actually be reached.
story-*/validate.py  ·  story-*/simulate.py
Delivery
Static files over HTTP. Two explicit modes: the local network, or the tailnet over real HTTPS. Progress lives in the browser — which is why it is per-device, and why there are no accounts.
Makefile  ·  manifest.webmanifest
author→ validate→ simulate→ serve

The pipeline is a real sequence: a chapter is not considered written until validate and simulate both exit zero.

03

One mechanism doing four jobs

A scene may grant an item. An item has a name, a description — and two flags that turn a simple inventory into the story's entire state machine: silent (never shown to the player, never announced) and persistent (survives the chapter boundary).

Cross those two booleans and you get every kind of state the stories need, out of a single mechanism the engine already had:

The eight silent-persistent “thread flags” in The Unwriting are how the story knows what kind of person you have been across nineteen chapters. It never says so. It just counts.

Choices carry the preconditions. Three gate types compose freely, and a fourth flag decides what a locked choice even looks like:

scene schema story-unwriting/ch10.json 
"war_council": {
  "id": "war_council",
  "text": "The map is bad and everyone knows it…",
  "item": { "id": "thread_told", "silent": true, "persistent": true },
  "choices": [
    { "text": "Say nothing.", "next": "east_road" },

    // gated on one item you must be holding
    { "text": "Show her the margin.", "next": "margin",
      "requires": "showed_margin" },

    // gated on a skill rank — hidden entirely until earned
    { "text": "[The Game 4] Rhee's gambit: stage the departure…",
      "next": "gambit",
      "skillRequires": { "game": 4 },
      "hideLocked": true },

    // gated on N-of-a-set: 6 of the 8 hidden threads
    { "text": "Tell her the truth about the standard.",
      "next": "end_one",
      "requiresCount": { "of": ["thread_queen", …], "min": 6 } }
  ]
}

hideLocked is the difference between a door you can see is locked and a door you never knew was there. The best ending in The Unwriting is the second kind — reachable only by having accumulated six of eight silent threads, and never advertised.

Why the chapter boundary is the good idea

At a chapter end, only persistent items, skills and skill points carry forward. Everything else is dropped. That is a narrative decision — chapters should feel like clean breaks — but it is also the decision that makes verification tractable: the story is nineteen small graphs chained together, not one graph with 2122 possible inventories. The design choice and the tractability are the same choice.

05

validate.py — the static pass

The first checker never plays the game. It reads the manifest and every chapter file and asserts things that must be true of the graph itself, walking the chapters in order and accumulating what the player could be holding by that point.

Structural
✓Every scene's id field matches the key it is filed under.
✓The manifest's startScene agrees with the chapter file's, and that scene exists.
✓No dead ends. A scene with no choices must be flagged end — otherwise it is a room the player cannot leave.
✓An end scene must not carry choices — the engine silently ignores them, so their presence means the author was wrong about something.
✓Every chapter has at least one end scene.
Referential
✓Every choice.next resolves to a scene that exists.
✓No unreachable scenes. A flood fill from the start must touch every node in the file — dead prose is caught the day it is written.
✓Every gated choice names items that actually exist by then — persistent items from earlier chapters, plus this chapter's own.
✓A requiresCount can never ask for more items than its pool contains.
Feasibility — the interesting one
✓A skill gate must be satisfiable at the moment it is drawn. Skill points arrive on a fixed schedule, so the checker knows the maximum rank bankable by any given chapter — and rejects a [Ledger 4] choice in a chapter where only three points have ever existed.
✓No soft-locks. Every non-end scene must offer at least one completely ungated choice, so no inventory can ever strand the player. (The final chapter's mutually-exclusive endings are the one deliberate exception, and simulate.py verifies it separately.)
Canon conformance
✓The waypoint budget from the bible is compiled into an assertion. The bible says chapter 12 awards five skill points; WAYPOINT_PLAN in the validator says the same; the chapter file must agree with both.
✓The eight hidden thread flags must be declared silent and persistent — a thread the player can see is a bug in the fiction.
✓Recap summaries must gate on real items, and the last one must be unconditional — a fallback that always matches.

Exit code 1 on any failure. The rule set is not generic: it encodes what this story promised to be.

The design document as executable specification

WAYPOINT_PLAN = {'ch3': 3, 'ch6': 4, 'ch9': 3, 'ch12': 5, 'ch15': 4, 'ch17': 3} is a line of Python that is also a line of the story bible. Twenty-two points across five skills, each capped at rank five — a budget of 22 against a capacity of 25. You cannot master everything, and that shortfall is the whole point of the skill system. It is enforced by arithmetic rather than by the author remembering.

Skill points, banked by chapter 22 awarded · 25 capacity · 5 skills × rank 5

Six waypoints, and never a seventh. A skill gate can only be written below the line — which is exactly what the feasibility rule checks.

06

simulate.py — playing every game there is

Static analysis proves the graph is well-formed. It cannot prove the story works — that the sixth ending is reachable by some actual sequence of choices, that no combination of items strands you, that every recap you wrote will ever be shown to anyone. Those are questions about reachable states, and there is only one honest way to answer them: play every game.

simulate.py is a second implementation of the engine's rules — items granted on entry, gates resolved the same way, persistent state carried across chapters — driven by an exhaustive depth-first search over (chapter, scene, inventory). This is model checking, pointed at a novel.

Done naively it does not terminate in any useful sense. With 122 item grants, the inventory alone spans 2122 combinations. Two abstractions make it finite:

Reduction 1 — only gate-bearing items are state

Most items never gate anything; they are props and colour. The simulator computes RELEVANT_IDS — the set of items some choice or summary actually tests — and keeps only those in the state key. A dosimeter you can carry but never use cannot change the future, so it is not part of the state. The rest of the inventory is, formally, irrelevant.

Reduction 2 — skills are bounded, not enumerated

Every allocation of 22 points across 5 skills is its own world, which is hopeless. So the simulator runs the entire story twice under two extremal policies: LOW, which never spends a point and therefore fails every skill gate, and HIGH, which assumes the best possible rank in whichever skill is being tested. Real play is bracketed between them. If the story is sound at both extremes, it is sound in between.

What comes out the other side, for The Unwriting:

Sixty-six times the search space, purely from unlocking skill-gated doors — a fair measure of how much story the skill system is actually holding open.

And the three things it asserts before exiting zero:

✓Every ending is reachable. All six terminal branches of chapter 18 — and their six epilogues in chapter 19 — are landed on by some concrete path. An ending that exists only in the file is a bug.
✓No soft-locks, empirically. Not "every scene has an open choice" (that was the static rule) but "no reachable state has zero available choices" — the real property, checked against real inventories.
✓Every conditional recap fires. If you wrote a summary that only shows when the player did X, some player must be able to do X — otherwise you wrote prose no one will ever read.
Where the guarantee stops

LOW/HIGH is an abstraction, not an enumeration. It brackets skill play; it does not visit every allocation, so a bug that needs exactly Ledger 2 and Cortex 3 simultaneously could in principle slip between the two policies. The soundness argument is that gates are tested one skill at a time, which HIGH's upper bound covers — but it is an argument, not a proof. Worth being precise about what has actually been shown.

Open finding — The Long Ellipse, chapter 4

!! ch4: summaries never matched at chapter end: [7] — a recap was written for a state no player can reach. Nobody has ever read that paragraph, and nobody can. It is dead prose, and the simulator has been saying so on every run. This is exactly the class of bug that is invisible to reading and obvious to a machine.

08

The runtime keeps its promises small

engine.js is a single object literal — config, state, DOM refs, and a set of methods that fetch a chapter, render a scene, type it out, and resolve choices. There is no reactive layer, no virtual DOM, no store. Rendering a scene is a function that appends paragraphs; the typewriter is a chain of setTimeout calls that can be cancelled by a click.

Progress is written to localStorage on every scene render — not on a save button — under a key derived from the story directory. Closing the tab mid-sentence loses nothing. It also means progress is per browser, per device: the same tailnet URL on a laptop and a phone are two separate save files. That is a real limitation, and an honest consequence of having no backend at all.

Serving is two Make targets, deliberately kept separate rather than one clever config: make serve-lan binds the local network interface, and make serve-tailscale binds loopback and lets Tailscale proxy it over real HTTPS to the tailnet, port included so it doesn't collide with anything else on 443.

The whole deployment story Makefile 
# local network — no Tailscale, no internet needed
serve-lan:
  python3 -m http.server $(PORT) --bind $$IP

# tailnet — loopback only, HTTPS terminated by Tailscale
serve-tailscale:
  python3 -m http.server $(PORT) --bind 127.0.0.1 &
  tailscale serve --https=$(PORT) 127.0.0.1:$(PORT)

A static site's deployment should fit on a napkin. This one does.

09

What the shape costs

Every one of these decisions bought something and gave something up, and the honest version of an architecture document says both.

No backend buys permanence — this runs from a file path, offline, forever, with nothing to patch and nothing to pay for. It costs cross-device sync, which is the one thing a reader actually notices missing.

Stories as data buys verifiability and a reusable engine. It costs expressiveness: anything a scene wants to do that the schema cannot describe has to become a new engine feature — which is precisely how skills and waypoints came to exist, and why they arrived in the nineteen-chapter story rather than the five-chapter ones.

A second implementation of the engine, in Python buys an independent check — the simulator disagreeing with the engine is itself a useful signal. It costs a synchronisation burden: two implementations of the same gate semantics, and nothing but discipline keeping them honest.

The chapter boundary buys a finite state space and a clean narrative beat. It costs long-range consequence — anything meant to echo across ten chapters must be smuggled through as a persistent silent flag, which is exactly what the eight thread flags are, and why there are only eight of them.
